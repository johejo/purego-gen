## Principles

- Use nix flake for reproducible development environment and build process.
- Use just for general task automation, such as running tests, linting, and building the project.
- Anayze C header files using libclang python bindings to extract function signatures, type definitions, constants, and other relevant information.
- Use uv for python package management. Do not execute python3 interpreter directly.
- Use basedpyrigh and pyrefly for strict type checking.
- Use ruff with ALL rules enabled for linting and formatting.
- Use pytest for testing framework.
- Prefer end-to-end golden tests for code generation, and unit tests for utility functions.
- Use jinja2 for code generation templates.
- Template files should be organized in a way that makes it easy to understand the structure of the generated Go code. Most logic should be implemented in the code generator, and the templates should be as simple as possible.
- All generated Go functions, types, constants, variables are prefixed with "purego_" to avoid name conflicts with user-defined code.
- Accept clang arguments from command line to support different platforms and configurations.
- Accept filtering options to allow users to specify which functions, types, constants, variables to generate code for.

## Command line interfece

```
# example usage
puregogen --header <header_file> --header <other_header_file> --symbol-filter <regex> --output <output_dir> -- <clang_args>
```
